Reference & Customization Guide

How Claude Decides Which Tool to Use

Understanding MCP tool matching — and how to control it

1. The Matching Mechanism

When you type a message to Claude and the Open Brain connector is enabled, Claude sees a list of tool definitions alongside your message. Each tool has three components that Claude uses to decide whether and when to call it:

1

Tool name — a machine identifier like search_thoughts. Claude considers this but relies more on the description.

2

Description — a natural language explanation of what the tool does and when to use it. This is the primary signal Claude uses to decide whether to invoke the tool. It's essentially a prompt to Claude about the tool's purpose.

3

Input schema — the parameters the tool accepts, with types and their own descriptions. Claude uses these to figure out what arguments to pass once it's decided to call the tool.

Key insight: The tool description is prompt engineering aimed at Claude. If you want Claude to invoke a tool more readily, you improve the description — the same way you'd improve a prompt. The description tells Claude "call me when the user says something like X."

Here's the decision flow Claude follows on every message:

A

Claude reads your message and all available tool descriptions

B

It determines whether any tool is relevant to your request — based on semantic matching between your words and the tool descriptions

C

If yes, it extracts the right parameter values from your message and calls the tool

D

It reads the tool's response and incorporates it into its reply to you

Why Claude sometimes doesn't call your tool: If the description doesn't clearly overlap with how you phrased your request, Claude will answer from its own knowledge instead. This is the #1 reason tools feel "unreliable" — the fix is almost always a better description, not a different prompt from you.

4. How to Customize Tool Descriptions

All tool definitions live in one file: 

~/open-brain/supabase/functions/open-brain-mcp/index.ts

Look for the TOOLS array. Each tool is an object with name, description, and inputSchema. Here's the anatomy: 

{
  name: "search_knowledge",          // Machine identifier — don't change
  description: "Search your...",      // ← THIS IS WHAT YOU TUNE
  inputSchema: {
    type: "object",
    properties: {
      query: {
        type: "string",
        description: "What you're..."  // ← Parameter-level hint
      },
      collection: {
        type: "string",
        description: "Optional:..."    // ← Also tunable
      }
    },
    required: ["query"]
  }
}

What to change

The description field is your primary lever. This is what Claude reads to decide if the tool is relevant. Think of it as a prompt aimed at Claude: "When the user says X, call this tool."

Parameter descriptions help Claude extract the right values from the user's message. If Claude is passing wrong parameters, improve these.

The name — don't change this unless you also update the handleTool switch statement.

Description writing tips

Be specific about when to use vs. not use. The description for search_thoughts says "For searching knowledge base documents, use search_knowledge instead." This disambiguation prevents Claude from calling the wrong tool.

List concrete examples. The search_knowledge description says "WhatsApp exports, blog posts, trading knowledge, podcast transcripts." These specific nouns give Claude more surface area to match against your prompts.

Mention languages. "Works in Hebrew and English" tells Claude it can pass Hebrew queries, which it might otherwise hesitate to do.

As you add collections, update the description. If you create a "Cooking Recipes" collection, add "cooking recipes" to the search_knowledge description so Claude knows to route food questions there.

Example: making a tool more trigger-happy

If you find Claude isn't searching your brain proactively enough, you could change:

// Before (passive — waits to be asked)
"Search your personal thoughts/captures by meaning."

// After (proactive — suggests when to use)
"Search your personal thoughts, captures, decisions, and notes by meaning.
Call this tool whenever the user asks about something they previously
thought about, decided, noted, or discussed. Also call it when context
from past thoughts would improve the answer — even if the user didn't
explicitly ask to search."

Caution: Making descriptions too aggressive causes Claude to call the tool on nearly every message, which slows down responses. Find the balance — specific triggers that match your actual usage patterns.

After editing, redeploy

cd ~/open-brain
supabase functions deploy open-brain-mcp --no-verify-jwt

Start a new conversation in Claude to pick up the changes (tool definitions are loaded at conversation start).

5. Designing for Your Knowledge Base Roadmap

As you add more collections (WhatsApp groups, blog posts, trading knowledge, podcasts), the key design decision is: how should Claude know which collection to search?

Strategy A: Let the user specify

Explicit routing — you mention the collection name and Claude passes it as a parameter.

You: "search trading knowledge: what's the consensus on NVDA options?"
→ search_knowledge(query="NVDA options", collection="Trading Knowledge")

Strategy B: Let Claude infer

Leave the collection empty and search everything. Good for when you don't know where the answer lives.

You: "what did anyone say about the new Apple headset?"
→ search_knowledge(query="Apple headset") // searches all collections

Strategy C: Update descriptions to route automatically

The most powerful approach. Add routing hints directly into the search_knowledge description:

"Search your knowledge base by meaning. Route by topic:
- Stock/trading/market/portfolio questions → scope to 'Trading Knowledge'
- WhatsApp/group discussion questions → scope to 'WhatsApp Groups'
- Blog/writing/content questions → scope to 'My Blog Posts'
- AI/LLM/GenAI/ML questions → scope to 'GenAI Podcasts'
Leave collection empty to search everything."

This teaches Claude to auto-route without you specifying the collection name. You'd update this as you add collections.

Recommended approach: Start with Strategy A (explicit) while you're building the system. Once you have several collections populated and know your usage patterns, evolve to Strategy C (auto-routing descriptions). Strategy B (search everything) is always available as a fallback.

6. Where Everything Lives

WhatWhere to editWhen to change
Tool descriptions open-brain-mcp/index.ts → TOOLS array When Claude picks the wrong tool or doesn't pick one at all
Parameter descriptions Same file → each tool's inputSchema.properties When Claude passes wrong values to a tool
Tool behavior (what the tool actually does) Same file → handleTool() switch cases When you want to change the output format or add logic
Database schema Supabase SQL Editor When you need new columns, tables, or indexes
Metadata extraction categories Same file → extractMetadata() system prompt When you want different type/category classifications
Embedding model Same file → model string in generateEmbedding() When you want a different model (keep dimensions at 1536)
Chunking logic Same file → chunkText() function When chunks are too big/small for your content type

The entire system is one TypeScript file, one database, and your descriptions. That's it. Every behavior you want to change is a string edit and a redeploy.

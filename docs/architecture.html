Open Brain Architecture

How your personal AI memory system works under the hood

System Overview

Open Brain is a personal knowledge system built on three free services. You type a thought — in Slack or through any AI assistant — and it gets automatically converted into a vector embedding (a mathematical representation of its meaning), classified with metadata, and stored in a PostgreSQL database. Any AI tool that speaks the MCP protocol can then search that database by meaning, not just keywords.

The Three Services

Supabase — your database & server

An open-source Firebase alternative. Gives you a PostgreSQL database, automatic REST API, and Edge Functions (serverless code that runs near your users). Your project is hosted at your-project-ref.supabase.co. The free tier covers everything you need.

OpenRouter — your AI gateway

A universal API that routes requests to any AI model. We use it for two things: generating vector embeddings (via OpenAI's text-embedding-3-small) and extracting metadata from your thoughts (via gpt-4o-mini). One key, one billing relationship, any model.

Slack — your capture interface

A private channel in your workspace (your-workspace.slack.com) where you type raw thoughts. A Slack app called "Open Brain" listens for messages and forwards them to your Edge Function. You can also capture directly from any MCP-connected AI.

Edge Functions Explained

Supabase Edge Functions are small pieces of server-side code (written in TypeScript/Deno) that run on-demand in the cloud. They don't need a server — they wake up when called and go to sleep after. You have two:

1. ingest-thought

This is the Slack integration. When you type a message in your #capture channel, Slack sends a webhook event to this function's URL. The function:

① Receives the Slack message and validates it's from the capture channel

② Generates a 1536-dimensional vector embedding of the text (via OpenRouter → OpenAI)

③ Extracts structured metadata — type, category, people, topics, action items (via OpenRouter → GPT-4o-mini)

④ Steps 2 and 3 run in parallel for speed

⑤ Stores everything as a single row in the thoughts table

⑥ Replies in the Slack thread with a confirmation summary

2. open-brain-mcp

This is the MCP server — the interface that lets any AI tool talk to your brain. It implements the Model Context Protocol as a JSON-RPC API over HTTP. When Claude (or ChatGPT, Cursor, etc.) wants to search your brain, it sends a JSON-RPC request to this function. The function processes it and returns results.

It exposes 6 tools that any connected AI can call:

ToolWhat it does
search_thoughtsSemantic search — finds thoughts by meaning, not keywords. Generates an embedding of your query and finds the closest matches using cosine similarity.
browse_recentLists recent thoughts chronologically. Can filter by type, category, or source.
brain_statsReturns totals and breakdowns — how many thoughts, by type, category, source, and people mentioned.
capture_thoughtSaves a new thought directly from any AI client — same embedding + metadata pipeline as Slack capture.
update_thoughtReplaces content of an existing thought by ID. Re-generates embedding and metadata.
delete_thoughtPermanently removes a thought by ID.

The match_thoughts function performs vector similarity search using the <=> cosine distance operator. An HNSW (Hierarchical Navigable Small World) index makes this fast even at scale.

How Semantic Search Works

Traditional search matches keywords — searching "career" only finds documents containing the word "career." Semantic search matches meaning. When you type "career changes," the system:

① Converts your query into a 1536-dimensional vector via the same embedding model

② Compares that vector to every stored thought using cosine similarity

③ Returns thoughts ranked by how close their meaning is to your query

④ "Sarah is thinking about leaving her job" matches "career changes" even though they share zero words

Authentication

The MCP server uses a simple access key passed as a URL parameter (?key=...), HTTP header (x-brain-key), or Bearer token. The key is stored as a Supabase secret and checked on every request. The database itself uses Row Level Security — only the service role (your Edge Functions) can read or write data.

MCP Protocol

The Model Context Protocol is an open standard (originally by Anthropic, now backed by OpenAI, Google, Microsoft, and the Linux Foundation) that lets AI clients discover and call tools. It uses JSON-RPC 2.0 over HTTP. The flow:

① AI client connects via GET (SSE) to discover the endpoint

② Client sends initialize → server returns capabilities

③ Client sends tools/list → server returns the 6 available tools

④ Client sends tools/call with tool name + arguments → server executes and returns results

Extending the System

The schema is designed for clean extension. Some paths forward:

New capture sources — Create another Edge Function (e.g. for Telegram, email, or a web form) that calls the same embed + classify + store pipeline.

Thought linking — Add a thought_links table with columns from_id, to_id, link_type to build a knowledge graph.

Tags — Add a tags table with a many-to-many join table for flexible labeling beyond the current topics array.

Swap models — Change the model strings in the Edge Functions and redeploy. Keep embedding dimensions at 1536 unless you also update the migration.

Dashboard — Build a React app that queries the thoughts table via Supabase's auto-generated REST API.

The architecture is deliberately boring. PostgreSQL, HTTP, JSON. No proprietary formats, no vendor lock-in. If Supabase disappeared tomorrow, you'd export a SQL dump and spin up Postgres anywhere. If OpenRouter changed pricing, you'd point at OpenAI directly. Every piece is replaceable. That's the point.
